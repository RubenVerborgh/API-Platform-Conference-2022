Web APIs provide access to
remote data or functionality.

  1. action-oriented APIs
    • RPC
    • SOAP
    • XML
    • …
  2. document-oriented APIs
    • REST APIs
    • LDP
    • …
  3. graph-oriented APIs
    • GraphQL
    • SPARQL
    • …

Graph APIs are just a specific kind of document APIs.

Common misconceptions prevent us from
designing effective knowledge graph APIs.

  1. Client abstractions require server abstractions.
  2. Querying requires dedicated query interfaces.
  3. One universal API can satisfy all use cases.

Client abstractions are not limited
to server-side abstractions.

  • Apps can be written using GraphQL
    without the server offering a GraphQL API.
  • GraphQL points to 2 distinct concepts:
    1. a read/write query language
    2. a Web API for processing such queries
  • Client libraries can provide GraphQL queries
    over non-GraphQL Web APIs.

The label query API is meaningless:
every API offers some kind of query.

  • Which API is a query API?
    1. /orders?id=5787
    2. /graphql?query={order(id:5787){product}}
  • Every API enables some kind of selection.
    • The difference is the granularity.
  • Client libraries can bridge the difference.
    • Execute queries over non-query APIs.

No single universal Web API exists
to satisfy all use cases.

  • The more expressive the query language,
    the more client-specific requests can be.
  • Web API design involves trade-offs.
    • Low expressivity leads to higher numbers of requests.
    • High expressivity leads to higher per-request cost.
  • Observations for single Web APIs
    don’t always generalize to multiple APIs.

On the decentralized Web of the future,
every person will have their own data pod.

  • Web 1.0 is a decentralized document space.
    • Documents link to other documents.
  • Solid brings standards for personal data pods.
    • Any piece of data can link to any other.
  • Multiple pods form a decentralized knowledge graph.
    • private and public parts
    • too large for any single system

Decentralized applications need
read/write access to a knowledge graph.

The graph is too big for any single place,
so apps reconstruct the whole via parts.

When knowledge fits in a single source,
we can place an API in front of it.

Abstract knowledge needs to be exposed
through concrete documents/resources.

A GraphQL API combines data from across
multiple resources of the “main” API.

Let's revisit the story for cases where
knowledge cannot fit in one place.

Each data pod exposes its part
as Linked Data through a Web API.

Different APIs for Linked Data
have different characteristics.

It quickly becomes unfeasible for apps
to deal with all possible API combinations.

A GraphQL Web API cannot help us,
since there's no server to attach it to.

The app can still be written with GraphQL:
a library translates it into HTTP requests.

Solid has kickstarted the transition
from API integration to data integration.

  • Solid app developers face challenges…
    …but API problems are not among those!
  • The way we write apps can be independent
    of the Web APIs offered by servers.
    • Clients can have their own abstractions.
  • Let's invest in reusable client development patterns.
